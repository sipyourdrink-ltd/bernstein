Bernstein vs Dorothy

Isolated task dispatch vs graph-based agent conversation for AI coding orchestration.

Dorothy is a graph-based multi-agent conversation framework where agents deliberate and pass messages over multiple turns until they converge on a result. Bernstein is a task dispatch system where agents get isolated tasks, work them independently in git worktrees, and exit after external verification. Dorothy is better when the problem requires multi-turn reasoning between agents. Bernstein is better when the problem decomposes into parallel independent subtasks.

What each tool is

Dorothy is a multi-agent framework built around conversation graphs. You define agents as nodes and message-passing edges. Agents can deliberate -- passing context back and forth over multiple turns until they reach a result. It is designed for problems where the reasoning process itself is the output: research synthesis, design deliberation, multi-perspective analysis.

Bernstein is a task dispatch orchestrator for CLI coding agents. It decomposes a goal into tasks, assigns each task to a short-lived coding agent (Claude Code, Codex, Gemini CLI), verifies the result against external criteria (tests, linter), and merges the output. The orchestrator is deterministic Python -- no LLM makes scheduling decisions.

The core architectural difference: Dorothy coordinates agents that talk to each other. Bernstein coordinates agents that do not.

When isolation beats deliberation

For coding tasks, isolation has specific advantages:

  • Hallucination containment. If Agent A takes a wrong turn and reasons incorrectly about an API, that incorrect reasoning does not propagate to Agent B and C. Each agent starts from the codebase, not from other agents' assumptions.
  • Parallelism. Independent agents can run simultaneously. Dorothy's conversation graphs have sequential dependencies by design.
  • Objective verification. "The tests pass" is a verifiable criterion. "The agents reached consensus" is not.
  • Speed. Multi-turn deliberation is slow. A Dorothy workflow with 3 agents doing 4 turns each is 12 LLM calls before any code is written. Bernstein dispatches coding agents immediately.

When deliberation beats isolation

  • Research and synthesis. "Analyze this codebase from a security perspective, cross-check with performance implications" benefits from agents challenging each other.
  • Ambiguous requirements. If agents need to negotiate what "done" means before starting, a conversation graph is better.
  • Multi-perspective review. Code review where a security agent, a performance agent, and a readability agent all comment on the same diff and respond to each other.
  • Problems without external verification. If there is no test suite and no objective completion criterion, agent consensus becomes the verification mechanism.

When to use Dorothy

  • The task requires agents to challenge and refine each other's reasoning. Research synthesis, design review, threat modeling, competitive analysis.
  • Requirements are ambiguous and need negotiation. Agents need to decide what to build before building it.
  • There is no external verification criterion. Success is "the output is good" rather than "the tests pass."
  • You want visible agent reasoning. Dorothy surfaces the full conversation between agents.
  • You are building an agent application, not automating coding tasks. Dorothy is a framework for building products.

When to use Bernstein

  • The task decomposes into parallel independent subtasks. REST endpoints + tests + docs can all happen simultaneously.
  • You need external verification. Tests either pass or fail -- agent consensus is irrelevant.
  • You want CLI agent support. 18+ adapters for installed CLI tools.
  • You want cost-aware model routing. Cheap models for simple tasks, expensive models for architecture decisions.
  • You want headless, overnight operation. bernstein --headless --budget 20.00 runs until the backlog is empty or the budget runs out.

Frequently asked questions

What is the difference between Dorothy and Bernstein for multi-agent coding?

Dorothy coordinates agents that talk to each other via conversation graphs and multi-turn message passing. Bernstein coordinates agents that work independently on isolated tasks with external verification. Dorothy is better for deliberation; Bernstein is better for parallel coding.

Is Dorothy or Bernstein better for parallel code generation?

Bernstein is better for parallel code generation because agents work independently in isolated git worktrees. Dorothy's conversation graphs have sequential dependencies by design.

When should I use conversation-based agent orchestration instead of task dispatch?

Use conversation-based orchestration (Dorothy) when the task requires agents to challenge and refine each other's reasoning, such as research synthesis, design review, or threat modeling. Use task dispatch (Bernstein) when the problem decomposes into parallel independent subtasks with external verification requirements.
