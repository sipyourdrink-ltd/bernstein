Bernstein vs Stoneforge

Provider-agnostic CLI orchestration vs provider-integrated IDE multi-agent coding.

Stoneforge is a provider-specific multi-agent coding framework with strong IDE integration and a polished developer experience through VS Code and JetBrains plugins. Bernstein is provider-agnostic and CLI-native, trading the IDE experience for model flexibility across 18+ agents, headless overnight operation, and deterministic scheduling that costs zero coordination tokens. The question is whether provider lock-in matters for your workflow.

What each tool is

Stoneforge is a multi-agent coding framework built around a specific AI provider's model. It runs multiple specialized agents within a structured session, with strong IDE plugin support and a first-party UI. The tight provider integration enables features like shared context across agents and provider-native tool use.

Bernstein is provider-agnostic. It wraps whichever CLI coding agent you have installed -- Claude Code, Codex, Gemini CLI, Qwen, or any CLI with a --prompt flag. The orchestrator is deterministic Python code. Agent sessions are isolated processes that exit after 1-3 tasks.

Cost comparison

Stoneforge costs are determined by the underlying provider pricing. If you use a single high-end model for all agents, costs reflect that.

Bernstein's bandit router reduces cost on low-complexity tasks by routing to Haiku or free-tier Gemini. For a mixed workload, the effective cost per task can be lower than running all tasks against a premium model. Benchmark data: median cost per task is $0.150 with 80% CI pass rate.

Provider lock-in: an honest assessment

Stoneforge's tight provider integration is a genuine advantage -- features that require deep access to provider internals (streaming token inspection, native tool use, session-level memory) work better with first-party integration. If you plan to stay with one provider indefinitely, this is a reasonable trade.

The risk: if that provider raises prices, introduces rate limits, or you want to use a newer model from a different lab, switching means rebuilding your orchestration layer. Bernstein's adapters abstract the provider interface -- swapping from --cli claude to --cli codex is one flag change.

When to use Stoneforge

  • You are committed to one provider. Policy, compliance, or trust model locks you to a specific vendor.
  • You want IDE integration. VS Code and JetBrains plugins show agent activity inline.
  • You need shared context across agents. Problems where agents need to read each other's output mid-task.
  • First-party support and SLAs matter. Commercial support from the vendor.

When to use Bernstein

  • You want provider flexibility. Mix Claude, Codex, Gemini, and Qwen in the same run.
  • You want no vendor dependency for orchestration logic. Task definitions, role templates, and janitor rules are plain files.
  • You want headless, overnight operation. CI pipelines, scheduled evolution runs, budget-capped overnight sessions.
  • You want self-evolution. Bernstein analyzes its own metrics and improves over time.
  • You prefer CLI-native tooling. No IDE extension required. Works in any terminal, including SSH.

Frequently asked questions

What is the difference between Stoneforge and Bernstein?

Stoneforge is a provider-specific multi-agent framework with strong IDE integration and shared session context. Bernstein is provider-agnostic (18+ CLI agents), CLI-native, and uses isolated git worktrees per agent with external janitor verification.

Does Bernstein work with VS Code or JetBrains IDEs?

Bernstein is terminal-native and does not have IDE plugins. It works in any terminal including SSH sessions and CI pipelines.

Is provider lock-in a real concern for AI agent orchestration?

If your provider raises prices, introduces rate limits, or a better model launches elsewhere, provider-locked orchestration means rebuilding. Bernstein's adapters abstract the provider interface -- swapping from claude to codex is a one-flag change.
