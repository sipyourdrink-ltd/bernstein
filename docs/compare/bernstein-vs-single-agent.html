Bernstein vs Single Agent

Multi-agent orchestration vs running one coding agent. Benchmark data from 25 real GitHub issues.

Running a single coding agent like Claude Code or Codex is the right call for simple, well-scoped tasks that take under five minutes. Bernstein exists for cases where a single agent bottlenecks on sequential work, lacks external verification, or needs to run overnight without supervision. Benchmark data on 25 real GitHub issues shows multi-agent Bernstein achieves 80% CI pass rate versus 52% for single-agent, with a modest cost premium.

The actual question

Multi-agent orchestration adds overhead: task planning, worker spawn/teardown, result verification, merge coordination. That overhead is worth paying when:

  1. The problem decomposes into independent subtasks that can run in parallel
  2. You need external verification (test runner, linter) before trusting agent output
  3. The task runs unattended and you cannot supervise failures
  4. The work is too complex for a single context window

For a 2-minute fix with a clear scope, none of those conditions hold. Single agent is faster, cheaper, and simpler.

What causes the CI pass rate gap?

Three structural reasons single agents fail more:

  • No external verification step. A single agent marks the task done when it thinks it is done -- not when the tests pass. Bernstein's janitor runs pytest and the linter after every task.
  • Context window saturation on complex tasks. A single agent working a 7-file change accumulates context from wrong paths and self-corrections. Bernstein decomposes the task into subtasks with isolated contexts.
  • No second pass on assumptions. Single agents commit to an approach early. Bernstein's manager can assign a verification subtask to a second agent.

When single agent is clearly better

  • The task takes under 5 minutes. Orchestration overhead is 15-30 seconds of spawn time plus verification.
  • You are iterating interactively. Reading agent output and adjusting the prompt in real time.
  • The task is hard to decompose. Fundamentally sequential work where each step depends on the previous one.
  • You need the agent's full reasoning visible. Real-time chain of thought in the terminal.
  • You want zero infrastructure. One command, one process, done.

When Bernstein is better

  • The task decomposes into 3+ independent subtasks. Auth middleware, tests, and docs can all happen in parallel.
  • You need verified output. "It looks right" is not "the tests pass."
  • You are running overnight or in CI. bernstein --headless --budget 20.00 runs until complete.
  • You have mixed task complexity. Route simple tasks to cheap models, complex ones to expensive models.
  • You want a record. .sdd/ files log every task, cost, and outcome.

The honest answer

Use a single agent. Then, when you notice the CI failing after "done" tasks, or spending an hour rerunning a single long session because it drifted off track, or leaving work queued overnight that needs babysitting -- that is when Bernstein is the right tool. Most users start with single-agent Claude Code or Codex. Bernstein solves specific problems that emerge after you have been using those tools for a while.

Frequently asked questions

Is multi-agent orchestration better than a single coding agent?

For medium-to-high complexity tasks, multi-agent Bernstein shows 28 percentage points higher CI pass rate with approximately 24% cost premium. For simple, well-scoped tasks, a single agent is faster and cheaper.

Why does multi-agent coding have a higher CI pass rate?

Three reasons: external verification (janitor runs tests), context isolation (each agent starts clean), and decomposition (complex tasks are split into simpler subtasks).

When should I use a single coding agent instead of Bernstein?

Use a single agent when the task takes under 5 minutes, you are iterating interactively, the task is hard to decompose, or you want zero infrastructure overhead.
